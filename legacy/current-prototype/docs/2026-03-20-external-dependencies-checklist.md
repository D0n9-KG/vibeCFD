# 潜艇智能体项目当前已完成工作与外部待补内容说明

这份文档只回答两件事。第一，当前工作区已经完成了哪些工作，而且这些工作目前完成到了什么程度。第二，如果项目要从“比赛展示版 demo”继续走向“真实可用系统”，还需要哪些外部能力、资源和约束补进来。文档不再展开本地还能继续做什么，只聚焦当前结果和外部缺口。

## 当前已经完成的工作

当前项目已经形成了一套可以本地启动、可以完整演示、并且足以支撑比赛展示的工作流系统。前端、后端、执行器、运行目录、事件历史、尝试记录和产物链路都已经打通。它不再是若干独立页面和几个临时接口的拼接，而是一套能够围绕一个 `run` 组织全流程的系统。

前端部分已经不是简单表单页，而是中文仿真工作台。用户可以在同一界面内完成任务输入、候选案例查看、历史运行切换、流程确认、图形结果查看、事件回放、产物浏览和报告阅读。工作台布局已经固定为顶部操作带、左侧工作栏、中央图形区、右侧检查器和底部 Dock，这使得整套系统具备了较强的软件工作台观感，而不再只是“AI 页面”。

后端部分已经完成了任务主链路。当前系统支持任务创建、案例检索、推荐流程生成、人工确认、排队执行、取消、重试、运行详情查询、事件历史查询和 attempt 历史查询。每个任务都会生成独立的 `run` 目录，后端会把当前状态、时间线、事件、attempt 以及最终产物都写入其中，并支持服务重启后的状态恢复。

执行层已经从单纯的 stub 模拟器升级成了项目专用主控执行器。它保留了独立服务边界，通过结构化执行请求 `/api/execute` 接入后端调度系统。执行器当前已经支持真模型调用的运行时形态，能够读取结构化任务上下文、理解选中案例与工作流摘要、调用受控 skill、生成几何检查结果、输出演示级求解结果，并最终生成比赛展示所需的中文 Markdown 报告和结构化产物。

围绕真实输入文件的处理能力也已经具备。桌面上的 `suboff_solid.x_t` 属于 Parasolid 文本格式，`suboff_solid.stl` 属于三角网格格式，当前执行器已经能够解析这两类文件，并将其映射到 DARPA SUBOFF 几何家族。针对 `.x_t`，系统可以读取 `KEY=suboff_solid`、应用来源、格式等头部元数据；针对 `.stl`，系统则会统计三角面、计算包围盒，并在尺度明显像毫米时自动折算到米级。基于此，系统已经能够生成几何概览图、压力分布图、尾流切片图、阻力表格、结果 JSON 以及最终报告。这意味着比赛展示时不再只是上传一个文件后生成几条占位文案，而是真的能围绕真实几何输入产生完整的 run 结果。

系统的持久化和可追踪性也已经成型。每个 run 会保留 `run_state.json` 当前快照、`events/events.jsonl` 结构化事件流，以及 `attempts/attempts.json` 执行尝试历史。服务重启后会自动恢复这些状态；如果某次执行在运行中被中断，系统也会识别为失败 attempt，而不是无声丢失。这对于比赛演示和后续答辩都很重要，因为它说明系统不是一次性“演给你看”的伪闭环，而是真的具备基本任务系统特征。

最后，工程化基础也已经补齐到适合演示的程度。仓库已完成 `.gitignore` 清理和 Git 基线提交，支持 `docker compose up --build` 的多服务本地部署，同时提供 `start-demo.ps1` 和 `scripts/run-desktop-demo.ps1` 两条启动路径。后者能够直接使用桌面的 `suboff_solid.x_t` 或 `suboff_solid.stl` 自动提交任务、自动确认流程并轮询至完成，从而快速生成一套可用于比赛展示的结果目录与报告。

## 仍然需要外部补进来的内容

虽然当前系统已经足以完成比赛展示，但如果项目要继续向真实工程系统推进，仍然有几类能力只能由外部补齐，当前工作区无法凭空生成。

第一类是更正式、更稳定的模型服务入口。当前执行器已经支持兼容 OpenAI 协议的模型调用方式，因此在工程层面并不依赖某一个特定品牌。但真正用于比赛或后续部署时，仍然需要外部明确最终可用的国产模型服务、访问方式、配额与鉴权策略。例如到底是使用阿里云百炼、DeepSeek、智谱还是企业内部封装过的兼容服务，这些都属于外部输入。当前工作区做完的是“执行器已经能吃下这类模型服务”，但具体用哪一套服务、哪一个模型以及对应的密钥策略，必须由外部决定并提供。

第二类是真实的 OpenFOAM 求解环境。当前系统已经具备求解边界、求解输入摘要、日志落盘和结果输出的完整位置，也已经有 `openfoam-adapter` 服务位。但这仍不等于真实 OpenFOAM 已经接入。要真正完成工程仿真，还需要外部提供可运行的 OpenFOAM 镜像或环境、固定命令入口、依赖工具链、典型成功样例、资源要求以及求解结果约定。当前项目的价值在于它已经把这些内容应该接入的位置和目录结构固定下来了，但求解器本体仍必须由外部交付。

第三类是真实案例资产和 benchmark 数据。当前案例库已经结构化，但更多承担的是“支撑当前 demo 工作流”的作用。若要进一步提升系统可信度，就必须由领域侧补齐高质量 benchmark 案例，包括几何家族、工况标签、推荐 solver、推荐 mesh、边界条件建议、参考结果、参考来源以及复用角色。没有这些资产，案例映射层虽然结构正确，但仍然主要是演示版知识库。

第四类是领域验收标准。系统已经能生成图、表、结果 JSON 和 Markdown 报告，但“什么样的结果算通过”“哪些图表是必须项”“报告里哪些章节是比赛必须展示的”“误差允许到什么程度”这些问题都不能由代码单方面定义。尤其是当项目从展示走向真实验证时，外部领域团队必须给出一套清晰的验收口径，否则系统很难判断自己是否已经真正达到目标。

第五类是更完整的专业工具层。当前执行器虽然已经有受控 skill 边界，也能生成演示级几何检查和结果产物，但如果未来要扩展到真实工程任务，仍然需要外部补进真实的几何处理工具、求解控制工具、后处理工具和报告工具，最好按能力域来交付，而不是堆积零散脚本。当前工作区已经具备这些工具未来接入的位置与调用边界，但工具本体本身并不在仓库里。

第六类是最终部署环境与权限边界。当前项目已经支持本地 Docker 化演示，但如果未来要进入比赛现场机器、实验室服务器或共享环境，仍然需要外部明确网络策略、容器运行条件、模型密钥管理、日志留存要求以及资源上限。代码可以适配这些条件，但无法替外部决定这些条件本身。

## 当前阶段结论

当前项目已经不是概念阶段，而是一套可直接用于比赛展示的工作流系统。它已经能处理真实几何输入，已经能形成完整 run，已经能通过主控执行器生成几何概览、压力分布、尾流图、阻力表、结构化结果和最终报告。换句话说，当前工作区已经把“系统骨架”和“比赛展示链路”都搭好了。

真正还缺的，不再是“本地再多写一点页面或接口”，而是若干只能从外部补进来的关键能力：正式模型服务、真实 OpenFOAM 环境、高质量 benchmark 资产、领域验收标准、专业工具实现以及最终部署边界。只要这些外部输入按当前边界接进来，现有系统就可以继续沿着同一条主线向真实可用系统推进，而不需要推翻重做。
