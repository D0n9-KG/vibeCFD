# 比赛演示版主控执行器设计

## 目标

这一版不再把执行器当成一个占位的 `claude-executor stub`，而是把它升级成一个面向潜艇外流场任务的项目专用主控智能体。它的定位不是通用 AI 编程代理，而是一个能够读取当前 `run` 上下文、理解上传几何与案例映射结果、在受控 skill 集合内推进任务、生成结构化产物并整理最终报告的执行服务。整个系统仍然维持现有的工作台前端、FastAPI 后端、调度器和 `runs/<run_id>/` 落盘体系，但执行层会从“模板化模拟器”改成“真模型调用 + 本地工具链”的混合架构。

第一版的标准不是“能力尽可能多”，而是“围绕比赛展示链路完整、稳、可解释”。用户能够上传真实几何文件，系统能够完成案例检索与流程确认，随后由主控执行器真实调用国产优先的兼容模型接口，按技能边界完成几何检查、求解准备、结果综合、报告生成，并把整套过程以时间线、事件、尝试记录、产物清单和最终 Markdown 报告的形式回写到前端和运行目录中。

## 架构边界

前端、后端和执行器的职责保持分层。前端继续承担交互工作台的职责，负责任务输入、案例查看、执行进度、产物和报告展示；后端继续承担任务入口、案例检索、流程草案、排队与调度、状态持久化；执行器则升级为真正的“主控智能体运行时”，负责消费结构化执行请求、驱动技能调用、聚合结果并产出结构化执行结果。这样做的好处是比赛展示时可以明确讲清楚：系统中有一个独立的主控智能体服务，而不是把所有逻辑都埋进 API 服务器里。

为了兼顾后续迁移成本和当前代码基础，第一版会保留现有的 HTTP 执行协议 `/api/execute` 以及 `ExecutorTaskRequest -> ExecutorTaskResult` 这一对结构化边界，但会扩充请求上下文与执行器内部模块。执行器内部将拆成四层：模型调用层、主控编排层、项目技能层、产物写回层。模型调用层负责兼容国产优先的 OpenAI-compatible API；主控编排层负责构造 prompt、约束可用技能、解析模型计划并驱动后续动作；项目技能层负责做真实的本地处理，例如解析 `.x_t` 头信息、生成几何摘要、准备求解目录、生成演示级 CFD 结果；产物写回层负责统一写出日志、JSON、CSV、SVG、报告和清单。

## 第一版必须完成的能力

第一版必须支持真实模型调用，但不追求完全开放式代理。执行器会以“受控智能体”的方式工作：模型负责理解任务、选择技能、生成阶段说明与结论；技能负责执行本地可控动作；最终由执行器整理成固定协议。这样既保留“主控智能体是真的在跑”的核心卖点，又避免了通用代理自由发挥带来的不稳定性。

技能层第一版聚焦在比赛展示真正需要的能力上。后端已有案例检索逻辑，因此执行器不必重新做完整案例搜索，但需要理解和利用后端传入的选中案例与 workflow draft。执行器内部的技能将包括几何检查、求解方案构建、演示级求解结果生成、结果组织和报告生成。几何检查会识别 `.x_t`、`.stl`、`.step/.stp`，其中 `.x_t` 至少能够解析 Parasolid 头信息并输出几何元数据摘要；若是 STL，则额外计算点云范围或三角面统计，并尽量生成一个可展示的几何概览图。求解部分第一版不追求真实高精度 OpenFOAM 闭环，但必须形成真实的求解边界、真实的日志和高可信度的结果产物，以便后续能平滑替换成真实求解器。

## 结果与产物设计

第一版的关键不是让模型“说了一段话”，而是稳定地产出可展示证据。每次运行完成后，`runs/<run_id>/` 至少应包含：执行请求与响应清单、主控执行日志、几何检查摘要、求解输入摘要、求解结果 JSON、关键指标 CSV、主图和副图 SVG、最终 Markdown 报告以及 artifact manifest。这样前端图形区、底部 Dock、右侧检查器都能拿到足够饱满的内容，不再依赖占位文本。

桌面上的 `suboff_solid.x_t` 将作为第一版的真实验证输入之一。它是一个 Parasolid 文本格式文件，头部包含 `KEY=suboff_solid`、`APPL=unigraphics` 等元数据，因此几何检查技能可以稳定抽取这些信息，并将其映射到 DARPA SUBOFF 家族。若后续同时存在同名 STL，则可以进一步生成几何包围盒与更丰富的预览；如果没有 STL，系统仍然能够基于 `.x_t` 元数据、案例模板与演示级求解器生成完整的比赛展示结果。

## 错误处理与可靠性

这版虽然是演示系统，但必须按“接近可发布”的标准处理失败。模型 API 调用失败、返回非结构化结果、技能执行异常、产物写出失败、求解阶段超时，都应该以结构化失败返回给后端，并映射为运行事件、执行尝试结果和可读的错误摘要。执行器不应该静默失败，也不应该只在控制台打印报错。

模型调用方面，第一版优先支持国产兼容 API，通过环境变量配置 `base_url`、`model`、`api_key`。为避免绑定某一家厂商，执行器只依赖 OpenAI-compatible 接口约定。默认部署时可指向国内服务；开发测试时若仅有兼容型其他服务，也可替换，但用户可见的产品叙事仍然是“国产优先的主控模型入口”。比赛环境如果网络允许，则直接走真模型调用；如果模型服务不可达，则运行明确失败并在前端显示，而不是偷偷回退成旧 stub。

## 测试与验收

验收标准分三层。第一层是单元测试，覆盖几何解析、模型响应解析、技能执行和协议回写；第二层是服务级测试，覆盖执行器 API 和后端调度到执行器的完整请求链路；第三层是端到端演示验证，使用桌面上的 `.x_t` 文件提交任务、确认执行、轮询完成，并验证关键产物与报告全部落盘。只有这三层都通过，第一版才算达到“接近可发布、可直接比赛展示”的要求。
