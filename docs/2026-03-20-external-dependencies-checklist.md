# 潜艇智能体项目当前已完成工作与外部待补内容说明

这份文档只回答两件事。第一，当前这个工作区已经实际完成了哪些内容，而且这些内容已经做到什么程度。第二，如果项目要继续从“可演示 demo”走向“真实可用系统”，还必须由外部补进来哪些能力、资源和约束。文档不再展开讲本地还能继续做什么，也不再把实施建议单独拆出来，而是尽量把当前结果和外部缺口写清楚，方便直接对外说明。

## 当前已经完成的工作

截至现在，这个项目已经不是概念验证或者静态原型，而是一套真正可以在本地启动、运行和演示的工作流系统。前端、后端、状态持久化、后台调度、执行边界和运行目录都已经形成闭环。虽然真实 Claude、真实 OpenFOAM 和真实专业工具还没有接进来，但“这些能力未来应该接在哪一层、通过什么边界进入系统”已经基本固定下来了。

### 1. 前端工作台已经搭起来了

前端目前已经完成了一套中文仿真工作台界面，而不是普通的表单页或者 AI 聊天页。页面被组织成固定工作区：顶部操作带负责全局动作和状态，左侧负责任务创建、案例浏览和历史切换，中间负责图形和结果展示，右侧负责流程确认和关键摘要，底部 Dock 负责承载详细信息、时间线、事件、attempt 历史、产物和报告。

左侧现在保留了 `任务 / 案例 / 历史` 三个一级工作区切换，这一层已经稳定下来。用户默认进入 `任务输入`，需要时再切到候选案例或历史运行，而不是把三块内容同时堆在一列里。`任务输入` 本身已经回到更直接的纵向表单，避免二级按钮过多导致阅读拥挤。中间图形区保留了 `图形视图 / 结果对比 / 案例映射` 三种视图，右侧检查器则集中显示流程确认、阶段状态、关键指标和工具/产物摘要。底部 Dock 已经能统一承载流程详情、时间线、结构化事件、执行尝试、产物目录和最终报告。

从用户体验的角度看，前端已经能够完整支持这样一条演示路径：创建任务、提交任务、查看候选案例、查看推荐流程、确认执行、观察状态变化、查看历史 run、查看事件与 attempt 历史、查看产物和最终报告。虽然图形区目前仍然主要承载占位视图和结果预留位，但工作台的基本交互方式已经成型了。

### 2. 后端主链路已经打通了

后端当前使用的是 `FastAPI`，并已经接入一层轻量的 `LangGraph` 编排骨架。Web 提交任务后，系统能够创建 run、读取结构化案例库、生成候选案例、生成工作流草案、等待人工确认，然后再把任务推进到后台执行阶段。也就是说，现在已经有了一条从前端任务输入一路通到后端运行目录的完整链路，而不是一组互相独立的接口。

目前后端已经具备这些实际能力：创建任务、读取 run 详情、列出历史 run、确认执行、取消排队、重试运行、读取结构化事件历史、读取执行 attempt 历史，以及围绕不同执行引擎的统一调用入口。确认动作也已经不再是“确认后在请求线程里直接跑完”，而是先把 run 置为 `queued`，随后由后台调度器异步领取任务、推进状态、记录 attempt，再调用真正的执行边界。这个变化非常重要，因为它意味着系统的运行模型已经开始接近正式平台，而不是一次性同步 demo。

### 3. run 目录、状态快照、事件流和 attempt 历史已经完成

当前系统最值钱的一块基础工作，其实是 run 持久化。现在每个 run 都会拥有自己的运行目录，并把关键状态写成实际文件保存下来。当前状态会写入 `run_state.json`，关键生命周期事件会追加写入 `events/events.jsonl`，每次真实执行派发都会写入 `attempts/attempts.json`。这意味着 run 已经不再是“只存在于内存里的临时对象”，而是一个有目录、有状态快照、有事件历史、有执行尝试记录的持久实体。

服务重启后，系统会自动扫描 `runs/*/run_state.json` 恢复历史 run。如果某个 run 在重启前还处于 `running`，系统也不会假装它还在正常执行，而是会把它识别成一次中断的执行尝试，并要求后续通过重试继续。这套机制虽然目前还不算复杂，但已经把“历史恢复”“失败归因”“run 复盘”和“可演示可信度”最基本的一层打出来了。

### 4. 异步队列、调度器和取消/重试机制已经完成

当前后端已经形成了明确的运行状态链路。一个 run 在人工确认后会进入 `queued`，随后由后台调度器领取并推进到 `running`，最后根据结果进入 `completed`、`failed` 或 `cancelled`。排队中的任务已经可以取消，历史任务已经可以重试，而且每次执行都会生成独立 attempt 记录，不会把多次执行混在一起。

这套机制带来的直接结果是，当前系统已经具备最基本的“任务系统”气质，而不是只有前端按钮和后端接口。run 已经拥有生命周期，执行已经有异步边界，失败已经有 attempt 记录，取消和重试也已经进入了统一的状态模型。

### 5. 案例检索与流程草案已经完成第一版

当前案例库已经是结构化数据，而不是写死在页面里的几条展示文案。系统会根据任务类型、几何家族和输入上下文生成候选案例，并进一步整理出一份工作流草案。这个草案里会包含推荐摘要、默认假设、允许工具、所需产物以及阶段说明。虽然它目前仍然主要依赖本地结构化规则，而不是真正的 LLM 检索推理或向量召回，但从系统架构上说，“案例驱动 workflow”这一层已经不再是口头概念。

这意味着当前仓库已经完成了从“用户说一个任务”到“系统给出候选案例和执行草案”的关键过渡。未来只要真实案例资产、真实模型推理和真实工具接进来，就可以在现有骨架上继续增强，而不需要推倒重来。

### 6. 执行边界已经预留出来了

当前系统已经不是只有一个“模拟执行”按钮。仓库里已经有三条清晰的执行边界。`mock` 引擎负责生成完整演示结果，用来证明日志、图像、表格、结构化结果和报告落盘都是通的。`openfoam` 引擎已经是一个求解边界层，能够接收输入并调用外部命令，只是暂时还没有接到真实 OpenFOAM 环境。`claude_executor` 则已经被做成独立执行器边界，当前仓库里带的是本地 stub 版本，用来先跑通“后端编排 -> 执行器服务 -> 结构化结果回传”这条链。

从项目推进的角度看，这一点非常关键。现在最重要的已经不是“真实 Claude 和真实 OpenFOAM 是否已经到位”，而是“等它们到位时，系统是否已经有明确的接入口”。这件事当前已经做完了，而且边界已经比较清楚。

### 7. 文档、启动方式和基础验证已经完成

当前仓库已经有完整的本地启动方式说明、Docker Compose 编排文件、执行器服务边界说明和外部接入说明。前后端测试、构建验证和 Compose 配置检查也都已经跑通过多轮。也就是说，这个项目不是一堆未经验证的拼接代码，而是一套已经被持续验证过的本地系统。

从“比赛展示”或者“向别人说明当前完成度”的角度看，当前工作区已经具备比较完整的交付基础。它可以演示，它可以讲清楚，它也能经受基本的本地验证。

## 还需要外部补进来的东西

如果项目目标只是停留在“可演示 demo”，那么当前工作区其实已经足够支撑说明和展示。但如果目标是继续往“真实可用系统”推进，也就是让 Claude 真正参与受控执行、让 OpenFOAM 真正完成 CFD 求解、让案例库真正承载工程知识，那么有几类内容必须由外部补进来，当前工作区本身无法凭空补齐。

### 1. Claude 的真实运行能力

当前仓库虽然已经有 `claude_executor` 边界、本地 stub 执行器、结构化请求协议和后端执行引擎，但这仍然只是“Claude 的接口位置和执行器外壳”已经就绪，并不代表 Claude 本身已经接进来了。要让系统真正进入“Claude 参与受控执行”的阶段，必须由外部提供真实可用的 Claude 访问方式。

这个外部输入可以是 Claude Code 运行时，也可以是 Anthropic API，也可以是企业内已经封装好的 Claude 调用服务，但无论是哪一种，都必须由外部提供认证方式、访问入口、超时与并发限制、日志要求，以及是否允许工具调用的规则。当前仓库缺的不是“再写一个 Python 文件”，而是一个真的能被 `claude-executor` 吃进去的模型运行环境。

### 2. OpenFOAM 的真实求解环境

当前仓库里的 `openfoam` 引擎已经把求解边界预留出来了，但真实的 OpenFOAM 运行远远不只是“执行一条命令”。它涉及 solver 版本、镜像或运行环境、入口脚本、依赖链、网格准备、求解监控、后处理工具、标准输出格式、结果目录约定，以及最关键的计算资源。这些都必须由真正拥有 OpenFOAM 环境的一方交付。

对当前项目来说，最有价值的外部输入不是一句“以后可以接 OpenFOAM”，而是一套可以直接适配的运行形态。例如固定镜像、固定命令入口、成功样例、输入输出约定、标准目录结构，以及首个可跑通 case。只要这套东西到位，当前工作区的边界层就能真正落地。

### 3. 真实案例资产和 benchmark 数据

当前案例库已经结构化了，但它更多承担的是“证明案例驱动骨架成立”和“支撑 demo 展示”的作用。要让系统真正具备工程意义，必须由领域侧补入真实案例资产。这些资产不应该只是几条标题或几份原始文件，而应该包括几何家族、任务类型、工况标签、推荐 solver、推荐 mesh、边界条件建议、参考结果、参考来源、预期产物，以及这个案例在新任务中的复用角色。

没有这些真实 benchmark 和案例资产，案例检索再怎么优化，也仍然只能停留在演示级别。当前工作区可以承载它们、组织它们、展示它们，但这些知识本身必须从外部来。

### 4. 领域验收标准

这是最容易被忽略，但实际上非常重要的一块外部输入。比如第一版究竟支持哪些任务类型，哪些结果算 MVP 范围，报告中哪些图表和字段必须出现，误差允许到什么程度，哪些 benchmark 可以作为可信对照，这些都不是写代码本身能决定的。如果没有这类验收标准，系统很容易陷入一种“技术上做了不少，但无法判断是不是已经达到目标”的状态。

当前工作区已经把流程、状态、产物和历史组织好了，但“什么叫完成”“什么叫可信”“什么叫可以答辩或验收”，必须由外部领域方来定义。

### 5. 最终部署环境与权限边界

如果后续系统不是只在本地机器上演示，而是要放到比赛现场、实验室服务器、共享环境或者云端，那么还必须由外部明确最终运行环境的条件。这包括模型访问方式、容器支持情况、网络策略、secrets 管理、日志留存要求、可用 CPU/内存/磁盘资源，以及权限边界。当前仓库可以继续把配置位和服务边界设计清楚，但最终运行环境的资源和安全约束，不可能由本地代码自己决定。

### 6. 专业工具层或 MCP 工具实现

当前系统已经有 `allowed_tools`、`required_artifacts` 这些工具和产物约束位，也已经有未来 MCP / skill 受控执行的接口位置。但真正的专业工具，例如几何检查、OpenFOAM 求解控制、后处理生成、报告整理等，如果希望以 MCP 或受控工具的形式让 Claude 使用，这些工具本体仍然需要由外部补入。当前工作区可以接它们、限制它们、记录它们，但无法凭空把专业工程能力变出来。

如果未来由外部来补这一层，最适合的方式不是塞进很多零散脚本，而是按能力域来交付，例如几何类工具、求解类工具、后处理类工具、报告类工具。这样当前仓库已有的受控执行边界和前端展示层才能真正发挥作用。

## 当前阶段的结论

当前这个工作区已经把最容易返工、也最值得先做的骨架打出来了。前端工作台已经能演示主要路径，后端已经具备 run 生命周期、持久化、事件历史、attempt 历史、后台调度和执行边界，文档也已经说明了真实接入应当落在哪些位置。也就是说，当前系统已经足够支撑展示、说明和继续迭代。

真正还缺的，不再是“再多写一点本地逻辑”，而是几类只能从外部补进来的关键能力：真实 Claude、真实 OpenFOAM、真实案例资产、验收标准、部署环境和专业工具实现。只要这些外部输入能按当前边界接进来，现有骨架就不需要推翻重做，而是可以继续在这条主线上往前推进。
