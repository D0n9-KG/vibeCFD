# 基于 DeerFlow 重建潜艇仿真智能体系统设计说明

## 1. 文档目的

这份文档定义接下来项目的重建方向。新的目标不是继续在当前轻量原型上叠加功能，而是直接以 DeerFlow 作为新的智能体底座，重新搭建潜艇仿真智能体系统。当前自建原型已经证明了比赛展示链路、几何输入处理、结果产物组织和工作台交互方向是成立的，但它的通用智能体能力仍然偏轻。DeerFlow 则已经具备更成熟的 `super agent harness` 能力，包括 `sub-agents`、`memory`、`sandbox`、`skills`、MCP、线程管理、artifact 管理以及对 Claude Code 的集成入口。因此，项目下一阶段的最佳路线不是继续维护两套并行框架，而是以 DeerFlow 为新主线，只把旧工程中真正有价值的领域资产迁入新架构。

这份设计文档同时定义两件事。第一，新的系统应该如何建立在 DeerFlow 之上。第二，现有工作区应该如何清理，以便后续开发不再被旧原型结构持续干扰。

## 2. 重建目标

重建后的系统仍然服务同一个产品目标：围绕潜艇外形几何与仿真任务，构建一套可以用于比赛展示、后续也能继续向真实工程系统推进的智能工作台。用户上传 `.x_t` 或 `.stl` 几何文件后，系统应当完成任务理解、案例匹配、流程确认、受控执行、结果生成、报告汇总以及完整的运行轨迹记录。

与旧版本不同的是，新的系统不再围绕“自写主控执行器”展开，而是以 DeerFlow 的 agent runtime 作为智能体内核。这样做的目标不是单纯追求框架时髦，而是让系统从一开始就拥有后续真正需要的能力，包括多 agent 分工、记忆沉淀、沙箱执行、技能系统、线程状态管理以及 Claude Code 集成能力。这些能力在当前比赛展示阶段可能不会全部显性暴露，但它们会决定系统后续是否具备继续扩展的上限。

## 3. 为什么选择 DeerFlow 作为新底座

DeerFlow 的定位与当前项目需求高度吻合。它不是单纯的聊天前端，也不是只提供单个 agent 的脚手架，而是明确拆出了 `Harness + App` 结构：底层 `Harness` 提供 agent 编排、skill 注入、memory、sub-agent、sandbox 和模型工厂；上层 `App` 提供 Gateway API、上传、artifact 管理和前端。这意味着它已经把当前项目中最通用、也最难自己打磨完整的那一层做成了独立基础设施。

对当前项目而言，DeerFlow 的价值主要体现在五个方面。第一，它原生支持多 agent 协作，这为后续把潜艇仿真任务拆成案例分析、几何检查、求解组织、报告汇总等多个角色提供了自然边界。第二，它有记忆系统，可以在长期运行中保留案例偏好、领域经验和用户设定，而不需要我们自己补一套轻量状态拼接逻辑。第三，它有 sandbox 与 artifact 模型，适合处理仿真任务里天然存在的大量输入文件、输出图像、结果表和报告。第四，它有成熟的 skills / MCP 扩展入口，后续可以更自然地接入专业工具。第五，它官方已经考虑了 Claude Code 等外部智能体能力的接入方式，这与我们之前希望把 Claude Code 放在主控链路中的思路高度一致。

因此，这次重建的核心不是“把 DeerFlow 接进旧工程”，而是“让 DeerFlow 成为新的工程骨架”。旧工程不再继续承担通用底座的角色。

## 4. 新系统的总体结构

新的仓库主线应采用 DeerFlow 现有的工程结构，并在其上增加潜艇仿真领域层。整体结构建议分为三层。

第一层是 DeerFlow 原生层，即它已有的 `frontend`、`backend`、`docker`、`config`、`skills` 和 LangGraph / Gateway 相关运行时。这一层不应该被大幅改造成“旧系统的样子”，而应该尽量保留其原生职责边界，方便持续继承 DeerFlow 上游的演进能力。

第二层是潜艇仿真领域层。这一层是新系统真正的产品差异所在，应该以新目录的方式明确收束，而不是散落在整个工程里。该层至少包含以下几类内容：潜艇案例资产、潜艇任务 skill、几何输入处理、结果产物组织规则、比赛展示报告模板，以及潜艇场景下的 artifact 命名与输出标准。换句话说，DeerFlow 负责“怎样运行一个复杂智能体系统”，潜艇领域层负责“这个系统在潜艇仿真场景下具体要做什么”。

第三层是旧工程归档层。当前轻量原型不会继续作为主线运行，但它依然包含早期有用的设计、界面、数据格式和处理经验。为了避免后续开发过程不断回头找旧代码，同时又避免误把旧代码当新主线继续演化，旧工程应整体归档到单独的 `legacy/current-prototype/` 目录中，只作为参考资料存在。

## 5. 新系统中应重点建设的能力

### 5.1 主控智能体与多角色协作

新的系统应直接利用 DeerFlow 的 `lead agent`、`sub-agents` 和 `skills` 机制，而不再保留当前自写的轻量执行器 loop。潜艇仿真场景至少可以拆出四类角色：任务统筹与流程决策、案例与经验检索、几何输入检查与预处理、结果汇总与报告整理。后续如果接入真实求解环境，还可以进一步引入求解控制角色和后处理分析角色。

这里的重点不是机械地“多搞几个 agent”，而是利用 DeerFlow 现成的角色体系，把潜艇仿真任务从单一 agent 的硬编码流程，升级为更清晰的角色分工体系。这样后续接入专业 skill 时，边界会更明确，也更符合比赛答辩时“智能体协作”这一类表述。

### 5.2 记忆系统

当前原型主要依赖单次 run 的状态与事件持久化，缺少跨 run 的经验沉淀。DeerFlow 的 memory 能力应被纳入新系统设计。第一阶段不需要把它做成复杂知识图谱，但至少应支持保留以下信息：常见潜艇几何家族与输入特征、历史案例的适用范围、推荐工作流偏好、报告撰写模板偏好，以及部分对用户任务有帮助的领域事实。这样系统在后续重复执行相似任务时，不会完全从零开始。

### 5.3 Claude Code 集成

新的系统应把 Claude Code 支持看作“内核能力之一”，而不是外挂脚本。DeerFlow 已经为 Claude Code 集成留出了模型配置和 provider 入口，因此新的潜艇系统应以 DeerFlow 兼容的方式接入 Claude Code，而不是继续使用当前原型里那套单独维护的轻量执行器协议。这样能让系统在需要时直接利用 DeerFlow 的 agent runtime、skills、sandbox 和 Claude Code 的能力组合，而不是人为切成两套不统一的执行链路。

### 5.4 潜艇领域技能

真正需要我们自己做的重点，是潜艇领域技能，而不是通用 agent 骨架。第一阶段建议明确以下技能边界：

- `submarine-case-search`：潜艇案例库查询、匹配和案例摘要生成
- `submarine-geometry-check`：`.x_t` / `.stl` 输入识别、尺度估计、家族映射和预览生成
- `submarine-solver-dispatch`：向后续 solver 或 OpenFOAM adapter 提交受控任务
- `submarine-postprocess`：对求解结果进行压力、尾流、阻力等结果整理
- `submarine-report`：生成比赛展示版中文报告、答辩摘要和 artifact 清单

这些 skill 是整个新系统最重要的领域资产。它们将决定 DeerFlow 在这个项目里不是一个通用研究 agent，而是一个潜艇仿真智能体平台。

### 5.5 前端工作台

DeerFlow 自带前端，但它的原生产品形态更偏 thread/chat/agent 交互。潜艇仿真场景需要的则是工作台化界面，包括几何视图、结果视图、指标面板、流程确认、历史任务和报告阅读。因此，新系统应采用“保留 DeerFlow 前端基础能力，重构产品界面”的方式。也就是说，线程、上传、artifact 管理、状态流可以直接承接 DeerFlow，但最终展示层应该演化成仿真工作台，而不是保留 DeerFlow 默认的通用 chat 产品外观。

## 6. 现有工作区中哪些东西应该保留

本次重建不是否定当前工作，而是要明确哪些成果值得继承。真正有必要保留的内容主要包括四类。

第一类是数据资产。当前仓库中的案例数据和 skill 元数据已经具备初步结构，可以作为 DeerFlow 新领域层的第一批输入。尤其是潜艇案例库本身，虽然还不够强，但比完全空白要好得多。

第二类是几何输入处理经验。当前原型已经具备 `.x_t` 和 `.stl` 的识别逻辑、尺度估计、简单几何分析和预览生成思路。这些逻辑不一定原封不动搬过去，但它们代表了当前项目对真实输入格式的早期适配经验，应该被视为领域能力参考，而不是被误当成旧框架的一部分整体丢弃。

第三类是结果产物组织规则。当前 `runs/<run_id>/` 的落盘思想、事件记录方式、报告与结果文件的组织方式，对比赛展示非常有价值。即使新系统切换到底层架构，也应保留“结果可追踪、artifact 可浏览、报告可直接展示”这一产品形态。

第四类是前端交互经验。当前工作台虽然仍然是演示版，但已经形成了一些正确方向，例如 3D 模型视图、结果图视图、左侧任务与案例、右侧检查器和底部结果台的布局思路。未来不一定复用旧代码文件本身，但这些交互结论应被保留。

## 7. 现有工作区中哪些东西应该停止继续投入

需要明确停止继续投入的，是当前自写的通用框架层，包括轻量主控执行器、轻量调度器、轻量运行时协议以及为弥补通用 agent 能力不足而临时补出来的各种桥接逻辑。这些东西在原型阶段是合理的，但在决定以 DeerFlow 为新底座之后，它们继续存在只会制造双框架并存的问题。后续开发若继续在这些模块上补功能，实际上是在增加未来要被废弃的技术债。

同样，当前根目录下大量围绕旧原型组织的 `backend`、`frontend`、`scripts` 和旧说明文档，也不应再以“主线项目”的身份存在。它们可以作为历史参考整体归档，但不应继续占据主目录、继续吸引注意力、继续被误认为未来开发主战场。

## 8. 工作区清理方案

工作区清理的目标不是“把旧东西删光”，而是明确主线和历史参考的边界，确保之后每次打开仓库，看到的就是 DeerFlow 主线，而不是旧原型。

清理后的仓库应遵循以下原则：

第一，DeerFlow 应上升为新的根目录主线。也就是说，当前从上游拉下来的 `tmp/deer-flow` 不应继续躺在临时目录里，而应成为仓库主结构的基础。

第二，当前项目中自建原型的核心目录，包括旧的 `backend`、`frontend`、`scripts`、旧 `README.md` 和原型说明文档，应整体移动到 `legacy/current-prototype/`。这样既保留参考，又不会和新主线混在一起。

第三，真正需要在新主线中继续使用的少量资产，应被显式抽取到新的领域目录，而不是让后续开发者到 `legacy/` 里临时复制。换句话说，`legacy/` 是历史档案库，不是运行时依赖。

第四，临时文件、测试日志、运行缓存和无价值的开发过程垃圾应直接清除，包括 `tmp-agent-*.log`、`tmp-backend-*.log`、`tmp-frontend-*.log` 以及其他一眼可见的本地调试残留。

## 9. 重建后的建议目录结构

建议以 DeerFlow 为主线后，将仓库整理为如下结构：

```text
backend/                     DeerFlow backend 与 harness 主线
frontend/                    DeerFlow frontend 主线，后续改造成仿真工作台
docker/                      DeerFlow 原生容器与部署支持
docs/
  superpowers/specs/         本次重建设计与后续实施文档
  submarine/                 潜艇项目自己的说明文档
domain/
  submarine/
    cases/                   潜艇案例数据
    skills/                  潜艇领域 skill 定义
    prompts/                 领域 agent / sub-agent prompt
    templates/               报告模板、结果模板
    schemas/                 结果与 artifact schema
legacy/
  current-prototype/         当前轻量原型整体归档
```

这个结构的关键，是让 DeerFlow 主线、潜艇领域层和历史原型三者职责清晰，不再混在同一平面。

## 10. 第一阶段实施重点

完成清理后，第一阶段不应该急着一次性重做所有功能，而是先把新系统的骨架真正立起来。第一阶段目标建议只有三项。

第一，完成 DeerFlow 作为新底座的本地可运行基线，确认模型、上传、thread、artifact 与 Claude Code / 兼容模型的接入路径可用。

第二，在 DeerFlow 新骨架里重新建立潜艇领域层的最小闭环，即能够接收 `.x_t` / `.stl` 输入、调用 `submarine-case-search` 与 `submarine-geometry-check`、生成最小结果摘要和中文报告。

第三，把 DeerFlow 前端逐步改造成潜艇工作台，而不是继续沿用通用 chat 体验。比赛展示对这个层面的感知最强，因此前端最终一定要体现出“专业仿真工作台”的产品气质。

## 11. 本次调整完成后的预期状态

本次工作完成后，仓库应达到如下状态：DeerFlow 已成为新的主线；旧项目已整体归档而非继续混用；潜艇领域资产的保留边界已经明确；后续开发的主方向转为“在 DeerFlow 上构建潜艇仿真应用”，而不是“继续扩写旧原型”。换句话说，仓库结构本身就应该清楚表达项目决策：我们已经不再维护一个自建轻量框架，而是正式切换到了更强、更系统、更适合长期扩展的 DeerFlow 底座。
