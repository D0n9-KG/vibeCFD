# VibeCFD 中文科研指挥台前端设计

**日期：** 2026-04-06

**状态：** 已根据前端重构讨论完成设计收敛，等待用户复审

## 1. 目标

本轮前端重构的目标不是继续修补现有工作台，而是在仅保留现有配色方向的前提下，彻底重做 VibeCFD 的前端信息架构、页面骨架、组件布局和交互语义，使其更接近“vibecoding 式的主智能体协作体验”，同时保留科研仿真场景所需的全过程透明度。

新前端必须让用户：

- 以中文界面使用整个产品
- 清楚看到从方案生成到最终报告的完整推进过程
- 始终能通过聊天框与主智能体协商、纠偏、追加要求或中止当前推进
- 在不丢失信息的前提下获得更简洁、更聚焦的界面
- 在 `submarine` 与 `skill-studio` 之间获得统一的产品语言与视觉体验

本轮重构是一次彻底的前端重构，而不是基于旧页面的渐进式换皮。

保留项只有：

- 当前配色与色调方向
- 后端能力覆盖
- 后端契约与数据来源

可完全重做的部分包括：

- 页面结构
- 信息架构
- 路由语义
- 组件体系
- 模块边界
- 导航组织
- 页面视觉骨架
- 中英文文案体系

## 2. 现状问题

当前前端即使已经做过一轮重构，仍然存在以下核心问题：

- 页面骨架仍然明显继承旧工作台，用户能感受到“左侧阶段轨 + 中间内容区 + 右侧 rail”的旧结构影子。
- `submarine` 仍然容易让用户觉得自己在浏览阶段，而不是在与主智能体共同推进一项研究任务。
- `skill-studio` 仍然更像功能仪表板，而不是技能资产生命周期工作台。
- 用户虽然能看到不少信息，但“方案、分工、技能、工具、执行、后处理、报告”之间缺乏一条清晰的主线叙事。
- 过多显式的控制条、状态块、阶段标签，会把体验拉回工程后台，而不是你要的科研协作工作台。
- 用户可见文案仍有较多英文残留，不符合中文产品要求。

因此，这一轮设计必须解决的不是“补更多卡片”，而是“彻底替换用户的主心智模型”。

## 3. 核心设计结论

本轮前端采用的目标形态是：

`中文科研指挥台 + 右侧协商聊天框 + 单主画布推进流 + 分层下钻证据面`

它不是阶段系统的外观优化，而是一个新的使用模型：

- 用户主要在中央主画布中理解“现在系统正在推进什么”
- 用户通过右侧聊天框直接与主智能体协商
- 主智能体负责根据聊天内容自主暂停、修正、改派、重跑、补充后处理或调整报告
- 过程细节、证据、日志、图谱、版本与发布信息通过抽屉和下钻层查看，而不是常驻占满主界面

## 4. 总体设计原则

### 4.1 主智能体优先，不再阶段优先

用户的主体验应当是：

`目标协商 -> 方案形成 -> 分工推进 -> 计算与后处理 -> 结论与报告`

而不是：

`阶段 1 -> 阶段 2 -> 阶段 3 -> 阶段 4`

后端内部即使仍保留阶段概念，前端也不应再把阶段轨作为主信息结构。

### 4.2 页面默认回答一个问题

中央主画布在任意时刻只回答一个主问题，例如：

- 现在方案推进到哪了
- 当前主要阻塞是什么
- 当前运行在做什么
- 当前结果是否足够支持结论

不能让一个页面同时平铺十几类同权重模块。

### 4.3 协商通过聊天完成，不通过显式控制面完成

用户已经明确要求：中断、修改、纠偏不需要复杂的按钮系统，只需要在聊天框里告诉主智能体即可。

因此前端不应再把下面这些做成主交互：

- `Pause`
- `Resolve`
- `Revise plan`
- `Dismiss notice`
- `No interruption is active`

取而代之的是：

- 右侧始终存在可展开的协商区
- 页面只保留轻量提示：用户可随时在协商区输入修改意见
- 主智能体收到后自行决定暂停、改方案、重派任务、重跑计算或调整报告

### 4.4 全流程可见，但不全量常驻

用户必须能看清：

- 方案生成
- 方案敲定
- 子代理任务分配
- skill 使用
- 实际计算流程
- 后处理方法
- 后处理结果
- 最终报告

但这些信息不应该永远全部展开。应采用：

- 主画布展开当前焦点
- 非焦点模块缩成摘要
- 详细过程通过抽屉、详情层、证据层查看

### 4.5 `submarine` 与 `skill-studio` 必须属于同一个产品系统

二者不是两套风格相近的页面，而应是同一个中文科研指挥台框架下的两种对象：

- `submarine` 面向研究任务
- `skill-studio` 面向技能资产

它们的配色、页面节奏、交互语言、协商方式、下钻方式应保持统一。

### 4.6 旧前端不是设计真相来源

旧前端和刚完成的第一轮重构前端，仅作为：

- 功能覆盖清单
- 后端契约参照
- 色彩来源

它们不构成以下内容的保留边界：

- 布局骨架
- 阶段组织方式
- 卡片组合关系
- rail 命名方式
- 页面分栏策略
- 路由与页面层级

## 5. 共享产品模型

前端应把每个线程统一理解为一个“主智能体协作会话”，无论它属于 `submarine` 还是 `skill-studio`，都具备以下共同结构：

- 当前目标
- 当前判断
- 当前推进状态
- 当前待确认事项
- 当前主要产出
- 过程叙事
- 证据与产物
- 用户可施加的修改入口
- 历史脉络与可追溯性

这意味着共享层不再是“阶段”，而是“会话状态 + 主画布焦点 + 协商入口 + 可下钻证据层”。

## 6. 共享页面骨架

## 6.1 顶部任务指挥条

所有工作台页顶部都使用一条固定的任务指挥条，用于展示最高优先级状态信息。

包含内容：

- 当前任务或技能名称
- 当前状态
- 主智能体的最新一句判断
- 待确认事项数量
- 最近一次关键动作
- 必要的全局操作入口

这条区域必须简洁，不能膨胀成第二套仪表板。

## 6.2 左侧轻导航

左侧只承担轻量导航职责：

- 在产品一级页面间切换
- 查看近期线程
- 回到新建入口

左侧不再承担主流程推进展示，也不承载高密度状态卡。

## 6.3 中央单主画布

中央是整个系统的核心。

设计要求：

- 始终只有一个主画布
- 主画布内部按当前状态展开一个焦点模块
- 其他模块以摘要形式折叠在同一条推进流中
- 不使用旧式常驻多阶段页签或阶段导航卡列

## 6.4 右侧协商区

右侧保留聊天框，但其角色从“附属聊天 rail”升级为“协商区”。

它的职责是：

- 接收用户修改意见
- 接收补充条件和新问题
- 让主智能体解释当前判断
- 允许用户要求停止当前推进并改动策略

它不应承担复杂控制台式按钮逻辑，也不应抢夺主画布注意力。

默认行为：

- 默认较窄，可展开
- 保持可用，但不喧宾夺主
- 作为用户唯一正式的协商入口

## 6.5 抽屉与详情层

对以下内容，不再常驻一整列面板，而应通过抽屉或详情层查看：

- artifact 浏览
- 日志
- 图谱
- tool 调用链
- 证据明细
- 版本历史
- 绑定关系
- provenance
- reproducibility
- experiment compare
- 发布闸门

## 7. 文案与语言规则

整套工作台的用户可见文案默认使用中文。

允许保留原文的仅包括：

- 模型名称
- skill id
- 文件名
- 版本号
- 必要的技术实体名

不应继续在主界面保留这些英文术语：

- `Define`
- `Evaluate`
- `Publish`
- `Graph`
- `Assistant Rail`
- `Negotiation Rail`
- `Pending approvals`
- `Show rail`
- `Hide rail`
- `Plan`
- `Execute`
- `Results`

应改写成更自然的中文科研工作台语义，例如：

- `任务定义`
- `验证与试跑`
- `发布与挂载`
- `关系网络`
- `协商区`
- `待确认事项`
- `展开协商区`
- `收起协商区`
- `方案推进`
- `计算执行`
- `结果与报告`

本条是硬约束，不是建议项。

## 8. `submarine` 工作台设计

`submarine` 应重构为“研究推进主画布”，不再是阶段工作台。

## 8.1 主画布核心结构

中央主画布以一条纵向推进流承载完整过程。

固定呈现以下模块顺序：

- 方案生成
- 方案敲定
- 子代理分工
- 技能调用
- 实际计算
- 后处理方法
- 后处理结果
- 最终报告

这八个模块始终可见，但默认只展开当前焦点模块，其余模块显示：

- 当前状态
- 一句话摘要
- 最近更新时间
- 是否可查看详情

## 8.2 模块含义

### 方案生成

展示：

- 主智能体提出的候选方案
- 关键假设
- 边界条件
- 目标指标
- 约束与风险

### 方案敲定

展示：

- 当前采用的方案
- 用户已确认项
- 尚待确认项
- 方案修订记录

### 子代理分工

展示：

- 已分派的子代理
- 各自负责内容
- 当前产出
- 是否完成或受阻
- 主智能体如何汇总这些结果

### 技能调用

展示：

- 用到了哪些 skill
- 为什么调用它们
- 输入输出边界
- 是否命中已有技能资产

### 实际计算

展示：

- 几何检查
- case 准备
- 求解器调度
- 运行状态
- 关键运行事件
- 中间产物摘要

### 后处理方法

展示：

- 使用了哪些后处理方法
- 为什么采用这些方法
- 对比策略
- 采用的指标和图法

### 后处理结果

展示：

- 核心图表
- 关键指标
- 异常点
- 不确定性
- 是否建议补算或对比试验

### 最终报告

展示：

- 报告草稿状态
- 可交付结论
- 结论边界
- 保留意见
- 后续建议

## 8.3 协商与打断

`submarine` 中不再设置显式“暂停流程”控制区。

正确的交互模型是：

- 用户直接在右侧协商区说“先停一下，方案要改”
- 或“先不要继续算，边界条件要改”
- 或“后处理方式换成压力分布对比”

主智能体收到后应：

- 停止当前推进
- 回到相应模块
- 说明已采用的修改
- 在推进流中留下变更记录

页面只需保留轻量提示：

`可随时在协商区输入修改意见，主智能体会停止当前推进并重新协商。`

## 8.4 `submarine` 下钻层

以下能力必须可见，但不应常驻主画布：

- 子代理执行详情
- skill 调用详情
- tool 调用链
- solver 执行日志
- 后处理方法详情
- artifact 浏览器
- 结果证据详情
- report package
- provenance 清单
- reproducibility 摘要
- environment parity 视图
- study / variant / compare 面板
- remediation 与 follow-up 历史

## 8.5 `submarine` 必须覆盖的后端能力

前端必须显式承载以下能力，不得只停留在隐藏 JSON 中：

- 方案生成
- 方案确认
- 子代理任务分配
- skill 使用轨迹
- 几何检查
- solver dispatch
- 实际计算流程
- 后处理方法
- 后处理结果
- 结果结论与 claim boundary
- evidence quality
- provenance
- reproducibility
- environment parity
- scientific delivery decision
- remediation handoff
- scientific follow-up
- 多 run / study / variant / compare
- 最终报告与报告产物

## 9. `skill-studio` 工作台设计

`skill-studio` 应重构为“技能生命周期主画布”，而不是技能仪表板或功能集合页。

## 9.1 主画布核心结构

`skill-studio` 与 `submarine` 共享同一套页面骨架与节奏，但主对象从“研究任务”切换为“技能资产”。

中央主画布建议固定呈现以下模块：

- 技能意图
- 技能草案
- 验证与试跑
- 发布准备
- 版本与回退
- 关系网络

同样采用：

- 当前焦点模块展开
- 其他模块摘要折叠
- 详细信息下钻查看

## 9.2 模块含义

### 技能意图

展示：

- 这个 skill 解决什么问题
- 给谁使用
- 在什么场景触发
- 成功标准是什么
- 由哪个 creator 或 assistant 创建

用户在线程开始前选择的创建者或助手模式，应在开始后继续可见，并作为会话 provenance 保存。

### 技能草案

展示：

- prompt 包
- 规则说明
- 输入输出约束
- 工作流步骤
- 示例
- anti-pattern
- 依赖资源

### 验证与试跑

展示：

- 结构校验
- 完整性校验
- 警告与错误
- scenario test matrix
- dry-run 证据
- 当前阻塞项

不能把这些信息压扁成单个 readiness badge。

### 发布准备

展示：

- readiness 状态
- 阻塞项
- 版本说明
- 显式绑定目标
- 发布风险

### 版本与回退

展示：

- 当前启用版本
- 当前已发布版本
- 当前工作版本
- 候选回退点
- 变更摘要

### 关系网络

展示：

- 上游依赖
- 下游复用
- 相似技能
- 高影响连接

默认只显示摘要，不占用整列主界面。

## 9.3 协商方式

`skill-studio` 与 `submarine` 共享同一条规则：

- 所有修改与打断都通过右侧协商区完成
- 不新增复杂控制按钮系统
- 用户直接告诉主智能体要改什么
- 主智能体自行回到合适模块继续推进

## 9.4 `skill-studio` 必须覆盖的后端能力

以下能力必须在新前端中有明确对应：

- skill 定义
- creator / assistant 选择
- workflow 结构
- examples 与 anti-patterns
- validation
- scenario-level tests
- dry-run evidence
- 发布阻塞项
- revisions
- bindings
- related skills
- 发布决策
- enabled state
- active revision
- 已发布版本
- rollback target
- version note
- explicit bindings

## 10. 共享的证据与信任面

以下信息属于信任关键面，必须在前端拥有命名化、可定位的展示入口，而不能退化为普通徽标或折叠说明：

- provenance
- reproducibility
- environment parity
- validation gate
- scientific claim boundary
- compare / study evidence
- dry-run evidence
- 发布闸门
- rollback readiness
- report package lineage

原则是：凡是影响“能否相信当前结论”“能否复现”“能否发布”“能否回退”的状态，都必须在前端被用户明确看见。

## 11. 视觉方向

视觉方向采用你已确认的：

`中文科研指挥台`

这意味着界面应当具有以下气质：

- 专业
- 克制
- 清晰
- 像研究任务控制台，而不是普通后台管理页
- 像与主智能体协作推进复杂任务，而不是翻流程卡片

保留：

- 当前配色与色调方向

自由重做：

- 版式
- 空间层级
- 卡片形态
- 字体组合
- 图标与标签语义
- 面板边界
- 页面密度策略

不应再追求：

- 大量 stage card 平铺
- 多列永久常驻信息墙
- 为了“显得复杂”而堆叠工程态控件

## 12. 路由与迁移方向

路由兼容不是设计边界。

如果新的工作台模型需要调整页面入口、嵌套路由或线程入口语义，允许直接调整。

推荐方向：

- `/workspace/submarine/new`
  - 进入新的中文研究推进工作台
- `/workspace/submarine/[thread_id]`
  - 进入同一工作台的线程态
- `/workspace/skill-studio/new`
  - 进入新的技能生命周期工作台
- `/workspace/skill-studio/[thread_id]`
  - 进入同一工作台的线程态

迁移时可以分阶段实施，但设计目标必须是一次清晰的前端重置，而不是长期保留旧式阶段工作台与新式主画布并存。

## 13. 非目标

本轮设计不追求以下方向：

- 与旧页面保持强视觉连续性
- 保留旧的阶段轨体验
- 保留英文工作台术语
- 把所有过程细节默认展开在主界面
- 把打断与协商做成复杂的操作台系统

## 14. 设计结论

VibeCFD 的前端应重构为一套真正统一的中文科研指挥台：

- `submarine` 用研究推进主画布讲清完整仿真流程
- `skill-studio` 用技能生命周期主画布讲清技能沉淀与发布流程
- 右侧聊天框作为唯一正式协商入口
- 过程、证据、产物、版本、图谱通过抽屉与详情层承载
- 页面只保留现有配色，不保留旧结构

这是最符合当前产品目标的方向，因为它同时满足了：

- vibe 式协作体验
- 全流程透明
- 用户随时可打断协商
- 中文产品表达
- 页面尽量简洁
