# VibeCFD 项目系统能力与后续优化总结

## 这份文档是写给谁看的

这份文档面向项目负责人、产品负责人、研发同学以及未来要接手这个系统的人。目标不是罗列代码文件，而是用中文把下面几件事讲清楚：

- 这个项目现在到底是什么
- 它现在已经能帮用户完成什么工作
- 一条真实链路从前端到后端到底是怎么跑起来的
- Skill Studio 和智能体链路之间是什么关系
- 现在离“科研生产级可交付”还差哪些关键能力
- 接下来最值得投入的优化方向是什么

本文基于 2026-04-16 当前工作区状态、已验证线程结果以及当前仓库实现整理。

## 一句话定位

`VibeCFD` 不是一个死板的 CFD 工作流脚本，也不是一个替科研人员拍板科学结论的“自动决策机器”。

它现在的定位更准确地说是：

**一个面向 CFD 任务的协作式智能工作台。**

用户在前端提出任务、上传几何、确认关键条件；系统在后台由主智能体协调几何预检、案例匹配、求解派发、报告整理、证据核对、后续修正建议等多个能力模块；最终把可追溯的结果、报告和后续动作再反馈到前端。

换句话说，这个项目当前真正想做的不是“替代科研人员”，而是“像 vibecoding 一样，成为科研人员做 CFD 时的强力搭档”。

## 目前已经成形的两条主链路

当前项目有两条已经建立起主干能力的链路：

1. `潜艇 CFD 链路`
2. `Skill Studio 链路`

这两条链路不是彼此孤立的。

Skill Studio 的目标，是把某一类专业能力沉淀成可复用 skill；而潜艇 CFD 链路则是这些 skill 真正被主智能体调度、被用户看到、被任务实际消费的场景。

所以这不是“一个做计算、一个做配置”的简单并列关系，而是：

- Skill Studio 负责沉淀和发布能力
- 主智能体负责在真实任务里调用这些能力
- CFD 工作台负责把这些能力组织成对用户可见的执行过程

## 先把几个容易看懵的词翻成白话

为了后面读起来更顺，这里先把文档里反复出现的几个词解释成更直白的话。

### `design brief`

可以把它理解成“这次 CFD 任务的结构化执行说明书”。

它回答的是：这次到底算什么、按什么条件算、算完希望交付什么。

### `solver dispatch`

可以把它理解成“把这次任务真正交给求解器去执行的阶段”。

它不是一句命令，而是一整套求解派发、运行、收集结果的过程。

### `artifact`

可以把它理解成“系统留下来的可查看结果物”。

报告、日志、摘要、JSON、HTML、验证结果，这些都属于 artifact。

### `scientific gate`

可以把它理解成“科研结论门禁”。

它判断的不是“有没有文件生成”，而是“当前证据够不够支持更强的科研说法”。

### `claim level`

可以把它理解成“这次结果当前允许说到什么程度”。

例如有时系统只允许把结果当成交付说明或工程参考，而不允许直接上升成高置信科研结论。

### `remediation`

可以把它理解成“如果当前证据不够，系统建议怎么补救、补跑、补验证”。

### `remediation handoff`

可以把它理解成“下一轮继续推进这条任务时要接住的交接单”。

它会告诉系统：之前卡在哪、下一步建议做什么、继续推进需要继承哪些上下文。

### `workspace` 和 `outputs`

可以把它们理解成两类不同用途的目录：

- `workspace` 更像计算现场，放真实运行中的 case、中间文件和后处理目录
- `outputs` 更像交付区，放整理后的报告、摘要、判断结果和用户要看的材料

## 一、从用户视角看：潜艇 CFD 链路到底做了什么

下面按“一个只看前端的真实用户”视角来解释。

### 1. 用户发起任务

用户进入潜艇工作台页面，新建一个线程，上传 `STL` 几何文件，并在聊天区输入任务描述。

这一步里，用户做的事情通常是：

- 上传一个潜艇或水下航行体的 `STL`
- 说明自己要做什么，例如阻力分析、基线工况、某个来流速度条件
- 指定是否希望立即执行，还是先做预检和确认

前端不是“上传完就后台偷偷跑完一切”，而是会把后续每个关键阶段都显式展示出来。

### 2. 系统先做几何预检，而不是立刻盲算

收到任务后，系统不会马上开算，而是先进入几何预检阶段。

几何预检主要解决的是：

- 这个文件到底是不是系统当前支持的输入格式
- 几何大概属于哪一类艇型或案例家族
- 尺度是否合理，是否存在毫米/米这样的单位歧义
- 是否能匹配到已有案例模板或 benchmark 背景
- 哪些关键参数还需要用户确认

这一步的价值很大，因为它避免了“输入错了、尺度错了、工况说不清，结果还继续往下跑”的假自动化。

### 3. 系统把“待确认项”明确展示给用户

预检完成后，前端不会只丢一句“有几项待确认”，而是会把待确认项作为可见信息呈现出来。

这类待确认项通常包括：

- 参考长度
- 参考面积
- 几何尺度归一方式
- 目标工况是否按当前理解继续
- 执行偏好是“继续生成设计简报”还是“直接准备执行”

这一层很关键，因为它体现了项目的核心产品思路：

**先协商，再执行。**

也就是说，系统不是黑箱自动化，而是可交互的协作式执行。

### 4. 用户确认后，系统生成结构化设计简报

当用户确认关键条件后，系统会生成一个 `CFD design brief`，也可以理解成“这次任务的结构化执行合同”。

这份设计简报不是给开发者看的内部垃圾数据，而是后续整个链路的中枢：

- 它记录了本次任务到底要算什么
- 它记录了几何和工况的当前确认状态
- 它记录了案例匹配结果和证据来源
- 它记录了后续预期交付物
- 它记录了当前能力边界和证据缺口

从这里开始，系统就不是只靠自然语言聊天来推进，而是把任务收敛成了可被工具和智能体稳定消费的结构化合同。

### 5. 系统派发求解，而不是直接输出一个“看起来像结果”的文本

设计简报确认后，系统进入 `solver dispatch` 阶段。

这个阶段做的事情不是“帮你写几句 OpenFOAM 命令”，而是把一次真实求解需要的关键对象准备出来：

- 生成 OpenFOAM 请求参数
- 生成实际 case 工作目录
- 生成运行脚本
- 启动受控求解
- 收集求解日志
- 提取结果摘要
- 组织验证与追溯信息

这里有一个很重要的设计边界：

**真实 OpenFOAM 中间文件放在 workspace，下游要看的交付产物放在 outputs。**

这样做的好处是：

- 研究运行目录不会和用户交付文件混在一起
- 真实计算中间文件可以保留给复现、排查和补充后处理
- 用户最终看到的是较干净、可解释的交付结果

### 6. 系统不是只给一个数，而是给一组可追溯结果

求解完成后，系统会产出一批结果，不只是一个阻力系数数值。

用户最终能拿到的结果通常包括几类：

#### 第一类：运行与求解执行结果

- 运行摘要
- OpenFOAM 请求参数
- 求解日志
- 求解结果 JSON / Markdown
- 网格统计
- 残差与稳定性证据

#### 第二类：研究与验证结果

- study plan
- study manifest
- 网格无关性验证结果
- 计算域敏感性验证结果
- 时间步敏感性验证结果
- 各个 study 变体的 run record 和 solver result

#### 第三类：最终交付结果

- 最终报告
- 交付就绪度判断
- 研究证据摘要
- scientific gate 结果
- remediation 计划
- remediation handoff

所以这个系统现在已经不是“帮你算完一个 baseline 数字就结束”，而是开始形成一套面向真实科研交付的结果组织方式。

### 7. 系统会明确告诉你“能交付什么”，而不是默认所有结果都能当科研结论

这是当前项目最重要、也最成熟的产品判断之一。

系统在报告阶段会区分：

- 这条链路软件上有没有跑通
- 这次计算在工程上是不是可复现
- 这次结果能不能直接支撑更强的科研结论

也就是说：

- 计算可以跑通，不代表论文级 claim 就自动成立
- 软件交付可以完成，不代表 benchmark 对齐已经充分
- 报告可以生成，不代表 scientific gate 一定放行

这是一种更诚实也更接近科研实际的系统行为。

### 8. 如果证据不够，系统会进入 follow-up / remediation，而不是假装已经结束

当系统判断当前研究证据还不够时，不会只停在“失败”。

它会继续给出：

- 当前被阻塞的原因
- 缺了哪一类验证证据
- 后续建议的修正动作
- 下一轮 follow-up 所需的 handoff 信息

这意味着系统已经初步具备“多轮协作推进同一 CFD 任务”的能力，而不是只支持一次性跑完。

## 二、从用户视角看：Skill Studio 链路现在能做什么

Skill Studio 不是装饰性的页面，它是这个项目能力沉淀机制的一部分。

它目前的目标是让团队可以把某一类能力做成 skill，并真正接入运行时。

### Skill Studio 当前链路的核心步骤

1. 用户在前端创建一个新的 skill 线程
2. 用户描述这个 skill 的用途、边界、适用对象和预期输入输出
3. 系统帮助生成 skill 草案
4. 系统组织测试/验证证据，而不是直接跳到发布
5. 用户或系统确认验证结果
6. skill 进入保存、打包、发布、安装状态
7. 运行时能够识别这个 skill 是否已可用、是否已安装

### 它为什么重要

因为这个项目并不想把所有能力都硬编码在主流程里。

真实的目标是：

- 主智能体负责理解任务和调度
- 专项能力沉淀成 skill
- skill 可以被测试、复用、发布、安装
- 未来不同领域、不同类型 CFD 任务都能沿着这条能力沉淀机制扩展

也就是说，Skill Studio 解决的是“系统如何持续长大”的问题。

### 它和潜艇 CFD 链路的关系

两者关系可以用一句话概括：

**潜艇 CFD 链路是能力消费场景，Skill Studio 是能力生产与治理场景。**

例如：

- `submarine-geometry-check` skill 帮系统做几何预检
- `submarine-solver-dispatch` skill 帮系统组织求解派发
- `submarine-report` skill 帮系统整理报告
- `submarine-orchestrator` skill 帮主智能体理解潜艇 CFD 任务该怎么推进

所以 Skill 不是孤立配置文件，而是未来整套多智能体能力系统的可治理单元。

## 三、从系统内部看：这套东西是怎么协同起来的

如果从架构视角看，当前系统大致可以理解成五层。

### 第 1 层：前端工作台

前端负责三件事：

- 让用户能发起任务、上传文件、继续 follow-up
- 把运行阶段、待确认项、证据、下一步动作可视化
- 把 artifact 和聊天历史串起来，让用户真正看见执行过程

当前这套前端已经不只是聊天窗口，而是一个带研究画布、状态轨道、操作面板、artifact 面板的协作工作台。

### 第 2 层：主智能体与线程状态

主智能体负责：

- 读取用户最新任务意图
- 决定是否先预检、先确认、先生成 brief、还是直接 follow-up
- 调用合适的内置工具和 skill
- 把结果回写到线程状态里

线程状态里最关键的一部分，是结构化的 `submarine_runtime` 快照。

它相当于这条任务链的“运行时真相”。

前端显示、后续 follow-up、工具执行衔接，都依赖它来保持一致。

### 第 3 层：领域能力层

这一层主要是潜艇 / 水下航行体 CFD 相关的领域逻辑。

它已经包含的核心能力包括：

- 几何预检
- 案例匹配
- 设计简报生成
- 求解派发
- 结果整理
- 验证组织
- remediation / handoff

这一层的价值在于把“会聊天的智能体”变成“知道 CFD 任务应该怎么推进的系统”。

### 第 4 层：受控执行层

这一层的职责是执行真实计算。

当前实现里，OpenFOAM 相关工作会进入受控 sandbox / case 工作区，系统会：

- 生成 case
- 执行 Allrun 或等价流程
- 保留日志
- 提取后处理结果
- 整理到结构化 artifact 中

这一步把自然语言协作真正落到了工程执行上。

### 第 5 层：artifact 与追溯层

整个系统很强调“结果不是一句话，而是一组有来龙去脉的交付物”。

因此每条线程下都会形成：

- 用户上传文件
- 结构化合同
- 执行记录
- 结果摘要
- 验证证据
- 报告
- remediation handoff

这让系统具备了比较强的追溯性和复现基础。

## 四、这个项目现在已经可以完成哪些真实任务

如果不夸大、不保守，当前项目已经能完成的事情可以分成三档。

### 第一档：已经明确打通的能力

#### 1. 从前端发起一个真实 STL 驱动的潜艇 CFD 任务

用户可以：

- 上传 STL
- 用自然语言描述目标
- 看到系统做几何预检
- 看到待确认项
- 确认后生成设计简报
- 触发求解派发
- 拿到结果和报告
- 在证据不足时继续 follow-up

这条主链路已经不是“理论上支持”，而是有真实线程和真实 artifact 支撑的。

#### 2. 用 OpenFOAM 执行一个真实 baseline case

当前项目已经能够围绕真实几何生成 OpenFOAM case 并执行，产出网格、日志、结果、报告。

以当前已经纠正过的 `DARPA SUBOFF` baseline 为例：

- 几何归一后参考尺度约 `4.356 m`
- 来流速度 `5.0 m/s`
- 当前有效结果 `Cd = 0.113173151`
- 当前有效结果 `Fx = 537.9440988 N`

这说明“从上传 STL 到拿到物理上合理数量级结果”的这条主干现在已经成立。

#### 3. 形成可追溯的最终报告与后续修正入口

系统不是只吐一个 `solver-results.json`，而是能继续形成：

- 最终阶段报告
- 交付判断
- 科学门控判断
- 证据缺口说明
- remediation handoff

这一步意味着系统已经开始具备“交付级组织结果”的能力。

#### 4. Skill Studio 的创建、验证、发布、安装状态链路

当前 Skill Studio 不只是生成一个 `SKILL.md` 草稿，而是已经具备：

- 生成 skill
- 展示测试证据
- 生命周期保存
- 发布
- 安装态识别
- 前端已安装状态可见

这说明“能力沉淀 -> 发布 -> 运行时可见”的机制已经有可用雏形。

### 第二档：可以用，但还不能过度承诺的能力

#### 1. 研究验证链条

系统已经能组织 mesh independence、domain sensitivity、time-step sensitivity 这些研究验证结构，但当前不能把它描述成“已经完全科研级闭环”。

原因不是这块完全没做，而是：

- 有验证框架
- 有 study 组织
- 有部分产物
- 但并不是每次都形成完整、可信、充分的研究结论闭环

#### 2. 科学结论分级

系统已经会给出 `delivery_only`、`blocked_by_setup` 等边界判断，这是成熟的方向。

但这套科学门控目前更像“诚实的守门员”，还不是“完整的科研审稿官”。

#### 3. 多轮 follow-up 协作

当前 follow-up 已经能接住上一轮 remediation handoff，继续刷新报告、继续推进任务。

但多轮复杂研究任务反复迭代时， lineage、状态压缩、长期证据组织还值得继续强化。

### 第三档：还不能说已经达到最终交付级的部分

#### 1. 论文级或高置信科研结论

当前系统已经能跑出工程上合理的结果，但还不能默认等价于“已经足够支撑高强度科研结论”。

特别是：

- benchmark 对齐工况不总是完全匹配
- 某些 sensitivity study 仍可能缺少完整结果
- 部分研究结论仍然停留在 `delivery_only`

#### 2. 更广范围的真实科研场景覆盖

当前主线最成熟的是潜艇 / 水下航行体方向，且输入主要围绕 `STL + OpenFOAM 模板化流程`。

如果要支持更多真实 CFD 科研场景，还需要进一步扩展：

- 更多任务类型
- 更多案例模板
- 更多 benchmark 库
- 更强后处理
- 更复杂几何与网格策略

## 五、一个当前最有代表性的真实例子：SUBOFF 基线任务

为了避免文档太抽象，这里用当前已经验证过的一条真实链路说明系统现在的实际水平。

### 这条任务做了什么

系统围绕一个 `DARPA SUBOFF` 实体 `STL`，完成了：

- 几何识别与尺度归一
- 案例匹配
- 生成设计简报
- 派发 OpenFOAM 求解
- 生成 solver result
- 生成最终阶段报告
- 生成 scientific gate 与 remediation 相关结果

### 这条任务当前拿到了什么结果

当前已验证的 baseline 结果包括：

- 参考长度约 `4.356 m`
- 速度 `5.0 m/s`
- `Cd = 0.113173151`
- `Fx = 537.9440988 N`

### 这条任务说明了什么

它说明三件事：

1. 系统已经能把真实 STL 走到真实求解
2. 系统已经能把错误尺度问题识别并纠正到合理结果
3. 系统已经能把“结果存在，但证据还不够强”的情况诚实体现在报告与 scientific gate 里

### 它也说明了什么还没完全到位

这条任务目前仍然提醒我们：

- baseline 可用，不等于科研结论链完全可用
- study variants 和 benchmark 对齐仍然是关键短板
- 报告已经比原来强很多，但距离真正“科研交付模板化”还有提升空间

## 六、用户最终能拿到什么，不只是“若干文件名”

如果把用户最终拿到的东西按意义来分，而不是按文件名分，可以分成下面几类。

### 1. 任务理解结果

这部分回答的是“系统理解我到底要做什么了吗”。

用户会拿到：

- 几何识别结论
- 案例匹配结论
- 待确认关键参数
- 结构化设计简报

### 2. 真实计算结果

这部分回答的是“系统到底算了什么、算到哪一步了”。

用户会拿到：

- 求解参数摘要
- OpenFOAM 运行痕迹
- 网格和残差摘要
- 力和系数结果
- 部分后处理或验证结果

### 3. 研究判断结果

这部分回答的是“这些结果现在能不能被当成强结论来用”。

用户会拿到：

- 交付就绪度判断
- 允许的 claim level
- 证据缺口说明
- scientific gate 状态

### 4. 下一步行动建议

这部分回答的是“如果现在还不够，下一步要做什么”。

用户会拿到：

- remediation plan
- remediation handoff
- follow-up 历史
- 建议补跑的验证或对比项

### 5. 复现与审计线索

这部分回答的是“以后别人如何复现或审计这次结果”。

用户会拿到：

- provenance manifest
- 运行记录
- 变体实验记录
- 报告引用到的关键来源路径

## 七、这些结果在目录里大致放在哪里

这里仍然解释一下目录，但重点是帮助理解，不是让你背文件名。

### 1. 用户交付结果

用户最终更关心的报告、摘要、交付判断，通常放在线程自己的 `outputs` 目录下。

可理解为：

- 这是“给人看、给后续协作看、给系统继续 follow-up 看”的结果区

### 2. OpenFOAM 中间运行文件

真实的 OpenFOAM case、脚本、postProcessing 等中间运行内容，通常放在线程自己的 `workspace` 目录下。

可理解为：

- 这是“给计算执行、复现和调试用”的工作区

### 3. 一个典型线程的组织方式

当前这类数据通常挂在：

- `backend/.deer-flow/threads/<thread_id>/user-data/outputs/...`
- `backend/.deer-flow/threads/<thread_id>/user-data/workspace/...`

这套分层方式本身就是当前项目的一个优点，因为它把“交付物”和“计算现场”分开了。

## 八、这个项目现在最值得继续优化的点

如果从“长期把它做成真正有战斗力的 VibeCFD 系统”来排序，下面这些方向最值得优先投入。

## 优先级 A：把科研任务的完整执行能力继续做深

这是当前最重要的方向。

原因很简单：项目的价值不在于前端页面多漂亮，而在于它能不能稳定完成真实科研任务。

这里最值得继续做深的点包括：

- 让 sensitivity study 真正稳定地自动跑完并被系统正确汇总
- 让 benchmark 对齐条件可配置，而不是靠人工额外解释
- 让比较结果、误差判断、接受阈值进入更稳定的合同结构
- 让 follow-up 能够自然触发补跑、重整理、再报告

这一层做深以后，系统才更像“科研执行伙伴”，而不只是“会组织一次 baseline 计算”。

## 优先级 B：继续强化主智能体与结构化合同之间的一致性

当前系统已经在往这条路上走，但还可以更彻底。

需要进一步优化的点包括：

- 减少聊天文本和结构化 runtime 快照之间的偏差
- 减少不同阶段对同一任务状态的重复表达
- 让 follow-up、report、scientific gate、artifact 展示共享同一份真相源
- 让老线程恢复时前端展示更稳定，不被陈旧状态污染

如果这一层不继续做强，项目越复杂，状态越容易乱。

## 优先级 C：把 Skill Studio 做成真正的能力治理系统

现在 Skill Studio 已经有创建、验证、发布、安装的雏形，但距离“能力治理系统”还有空间。

后续很值得继续做的点包括：

- skill 版本管理
- skill 适用范围声明
- skill 与主智能体策略的绑定关系可视化
- skill 测试结果标准化
- skill 发布后的回滚、替换、弃用机制

只有当 skill 真正可治理，系统才不会随着能力增加而越来越乱。

## 优先级 D：提升报告的科研交付质量

当前报告已经比早期版本强很多，特别是：

- 不再只是一串文件
- 已经开始分组组织交付文件、求解输出、工作区中间文件
- 已经开始表达证据边界与 claim level

但如果目标是面向科研人员长期使用，报告还值得继续提升：

- 增加更规范的章节模板
- 增加图表、曲线和关键截图
- 增加更自然的结果解释语言
- 增加“这次计算实际生成了哪些关键产物、它们分别有什么作用”的整理页
- 增加摘要版与审计版双视图

## 优先级 E：最后再系统性打磨前端信息架构

前端当然也重要，但以当前阶段看，它更像“承载真实能力的呈现层”，不是第一性问题。

等核心后端和智能体链路进一步稳定后，前端最值得优化的是：

- 信息分层，减少“所有信息同时堆给用户”
- 让每个阶段的“下一步动作”更清晰
- 让 artifact 展示从“文件名列表”升级成“用户能看懂的结果卡片”
- 让 follow-up、remediation、study 对比结果更直观

简单说，前端下一步最重要的不是“更炫”，而是“更清楚、更科研工作台化”。

## 九、如果用一句话评价这个项目当前所处阶段

如果必须给当前阶段一个准确判断，我会这样描述：

**它已经跨过了“概念原型”阶段，进入了“主链路可运行、能力开始成体系、但离科研生产级稳定交付还有关键深水区要攻克”的阶段。**

它已经不是一个只能演示几张页面或输出几段文本的壳子了。

它已经具备：

- 真任务入口
- 真几何预检
- 真结构化合同
- 真求解派发
- 真 artifact 追溯
- 真报告与 gate
- 真 follow-up
- 真 skill 生命周期主干

但它也还没有走到“只要交给科研人员就可以毫无保留地长期生产使用”的程度。

当前离那个目标最近、也最值得投入的，不是继续缝表面，而是继续把“复杂真实科研任务能否稳定协商、执行、验证、补跑、交付”这条主线做深做硬。

## 十、建议的下一阶段目标

如果要给下一阶段定义一个最清晰的目标，我建议是：

**把系统从“能完成一条代表性 CFD 链路”推进到“能稳定完成一类真实科研 CFD 任务族”。**

对应到执行上，可以拆成下面四件事：

1. 把 study / benchmark / follow-up 主线彻底做实
2. 把 runtime 状态与 artifact 组织再统一一层
3. 把 Skill Studio 的治理能力补齐
4. 最后再基于稳定主线反推前端信息架构优化

这样推进，项目会越来越像 `VibeCFD`；反过来，如果过早把重心放在外观或局部缝补上，系统会越来越像“页面很多、能力零散”的半成品。

## 结尾判断

当前这套系统已经能说明一件事：

**VibeCFD 这条路不是空想，主干已经搭起来了。**

接下来真正决定成败的，不是“还能不能再跑通一次 demo”，而是：

**能不能把复杂真实科研任务的协商、执行、验证、补跑和交付，持续变成系统的稳定能力。**
