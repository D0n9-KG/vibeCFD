# v1.1 DeerFlow Alignment Notes

**Researched:** 2026-04-03
**Milestone candidate:** `v1.1 Experience & Platform`

## Official DeerFlow Capabilities Relevant to v1.1

- DeerFlow treats skills as progressively loaded capability modules built from `SKILL.md` plus optional `scripts/`, `references/`, and `assets/`.
- DeerFlow distinguishes `skills/public/` from `skills/custom/`; custom skills are user-created or installed and are typically organization-specific.
- DeerFlow supports custom-skill installation through API, manual copy, or `.skill` archives, and skill enable or disable control through API and `extensions_config.json`.
- DeerFlow positions Docker deployment, sandbox execution, and Kubernetes provisioner mode as first-class deployment paths rather than afterthoughts.
- DeerFlow mounts skills into sandbox paths such as `/mnt/skills/custom/...`, so generated skill packages should be authored with sandbox-aware resource paths.

## Current Project Alignment

- The repo already contains a dedicated skill studio workbench, a submarine skill-studio domain module, publish and install API endpoints, skill graph helpers, and extensions-config writes.
- The repo already ships Docker definitions for `frontend`, `gateway`, `langgraph`, `provisioner`, and an OpenFOAM sandbox image.
- Sandbox configuration already exists in the DeerFlow config layer, and runtime-profile vocabulary has been introduced for provenance honesty.

## Gaps Observed

- The current skill studio behaves more like a draft and publish flow than a full lifecycle management product with revision history, rollback, or clear post-publish governance.
- Major workspace surfaces are still implemented in very large components, which makes UX iteration and architectural reasoning harder than it should be.
- Server-safe isolation exists in pieces, but the project still needs a more opinionated, least-privilege deployment baseline for lead agent, subagents, and OpenFOAM runtime execution.
- Live non-mock validation still trails deterministic mock validation in a few critical places.

## Planning Implications for v1.1

- Treat skill creation as a product surface, not a hidden artifact pipeline.
- Align custom Vibe CFD skills with DeerFlow's native skill format, publish mechanics, and sandbox paths.
- Use the milestone to turn Docker and sandbox support into a server-ready operating model with clear boundaries and health checks.
- Pair the frontend overhaul with deliberate module decomposition so UX improvement and architectural cleanup happen together instead of fighting each other.
